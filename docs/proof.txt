## A1) Risk-adjusted portfolio construction (not return-only)

Assumption
- Portfolio selection should optimize risk-adjusted outcomes under constraints (position/sector concentration and mandate settings), not raw return.

What the code does now
- Portfolio generation and analytics are centralized in `apps/api/app/services/db_quant_engine.py`.
- Frontend generation/analysis/backtest flows consume backend outputs via `src/services/backendApi.ts`.

Why this is kept
- Constrained optimization and risk-aware allocation are consistent with institutional portfolio practice.

Sources
- Markowitz, H. (1952), Portfolio Selection, Journal of Finance.
- Ledoit, O. & Wolf, M. (2004), Well-Conditioned Covariance Estimator, Journal of Multivariate Analysis.
- Grinold, R. & Kahn, R. (2000), Active Portfolio Management, McGraw-Hill.

## A2) Ensemble scoring with explicit degradation modes

Assumption
- Combining multiple model families is preferable to single-model dependence when artifacts are available, with explicit degraded/rules-only behavior when they are not.

What the code does now
- Ensemble scoring path remains active in `apps/api/app/services/ensemble_scorer.py`.
- Runtime/model state is exposed through `GET /api/v1/models/current` via `apps/api/app/services/model_overview.py` and route wiring in `apps/api/app/api/routes/models.py`.

Why this is kept
- Ensemble methods improve robustness to single-model failure and regime instability.

Sources
- Dietterich, T. (2000), Ensemble Methods in Machine Learning.
- Zhou, Z.-H. (2012), Ensemble Methods: Foundations and Algorithms.
- Kuncheva, L. (2014), Combining Pattern Classifiers (2nd ed.).

## A3) Friction-aware backtesting

Assumption
- Backtests must include realistic cost/tax components and not rely only on gross return curves.

What the code does now
- Backtest APIs are served from `apps/api/app/api/routes/backtests.py` and engine calculations from `apps/api/app/services/db_quant_engine.py`.
- UI reports cost/tax fields in the backtest surface.

Why this is kept
- Gross-only backtests materially overstate realizable performance.

Sources
- Bailey, D. et al. (2014), The Probability of Backtest Overfitting.
- Novy-Marx, R. & Velikov, M. (2016), Trading Costs and Realizable Returns.
- Zerodha charges reference (India brokerage/tax schedule): https://www.zerodha.com/charges/#tab-equities

## A4) Market-calendar-aware ingestion scheduling

Assumption
- Ingestion logic should use exchange-aware calendar/session checks instead of weekday-only assumptions.

What the code does now
- `apps/api/app/ingestion/nse_bhavcopy.py` uses market-calendar helpers for trading-day selection.
- Scheduler lifecycle is managed in `apps/api/app/services/scheduler.py` and wired in `apps/api/app/main.py` startup/shutdown events.

Why this is kept
- Weekend-only logic is insufficient around exchange holidays/special sessions.

Sources
- NSE holiday calendar (official source): https://www.nseindia.com/market-data/holiday-calendar
- APScheduler user guide/API docs: https://apscheduler.readthedocs.io/en/3.x/

## A5) Rich route contracts for benchmark/market/trade-ideas

Assumption
- The API should return typed envelopes with runtime/notes/context where needed, not only minimal arrays.

What the code does now
- Benchmarks supports summary + compare (`apps/api/app/api/routes/benchmarks.py`).
- Market data supports summary + regime dashboard (`apps/api/app/api/routes/market_data.py`).
- Trade ideas supports list envelope + `/screen` + symbol detail (`apps/api/app/api/routes/trade_ideas.py`).
- Corresponding schemas are in `apps/api/app/schemas/portfolio.py` and `apps/api/app/schemas/trade_idea.py`.

Why this is kept
- Rich contracts reduce frontend guesswork and enable explicit runtime/fallback communication.

Sources
- OpenAPI specification: https://spec.openapis.org/oas/latest.html
- FastAPI docs (typed response models and OpenAPI generation): https://fastapi.tiangolo.com/
- Espinha, T. et al. (2014), Web API Growing Pains.

## A6) Mandate semantics include explicit risk mode mapping

Assumption
- User risk attitude should map to deterministic engine risk modes to keep portfolio behavior explainable and reproducible.

What the code does now
- `apps/api/app/services/mandate.py` maps `risk_attitude` to `risk_mode` (`ULTRA_LOW` / `MODERATE` / `HIGH`) used downstream in engine logic.

Why this is kept
- Deterministic mapping supports reproducibility and auditability of generated portfolios.

Sources
- Fabozzi, F. et al. (2007), Robust Portfolio Optimization and Management.
- Meucci, A. (2005), Risk and Asset Allocation.

## A7) LLM explanation is secondary, not primary decision logic

Assumption
- Generative explanation should remain auxiliary to quant outputs and never be the core trading decision engine.

What the code does now
- Stock detail explanation path exists in `apps/api/app/api/routes/stock_detail.py`.
- Core generation/scoring paths remain in quant services; no chat tab is active in current UI.

Why this is kept
- Hallucination and non-determinism risks are better contained when LLM is explanatory.

Sources
- Bender, E. et al. (2021), On the Dangers of Stochastic Parrots.
- Ji, Z. et al. (2023), Survey of Hallucination in Natural Language Generation.
- NIST AI RMF 1.0 (2023): https://www.nist.gov/itl/ai-risk-management-framework

## A8) Single frontend API adapter for contract parity

Assumption
- A centralized frontend API adapter reduces drift and inconsistent handling across tabs.

What the code does now
- `src/services/backendApi.ts` is the primary typed mapping layer used across Overview/Market/Portfolio/Ideas/Backtest/Compare.

Why this is kept
- Centralizing mappings and fallback behavior lowers integration breakage risk.

Sources
- Maleshkova, M. et al. (2010), Investigating Web APIs on the WWW.
- Espinha, T. et al. (2014), Web API Growing Pains.

## A9) Current product-surface assumptions

Assumption
- The maintained workflow is limited to current tabs and route families; removed features should not be documented as active.

What the code does now
- Active tabs: Overview, Market, Portfolio, Trade Ideas, Backtest, Compare.
- Removed from active UI: chat/events/rebalance surfaces.

Why this is kept
- Documentation and smoke coverage should mirror shipped behavior, not legacy experiments.

Sources
- FastAPI route registration in `apps/api/app/api/router.py`.
- UI tab registration in `src/App.tsx`.
